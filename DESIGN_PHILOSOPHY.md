# OasisWaker 设计哲学

> 版本: 1.0 | 最后更新: 2026-07-13

---

## 核心原则：协议层 vs 适配器层

OasisWaker 既不是"别人适配我们"，也不是"我们适配别人"。它是一套**分层设计**：

```
┌──────────────────────────────────────────────┐
│  用户侧 CLI / Dashboard / REST API           │
│  (OasisWaker 协议 — 块存储/加密/注册/心跳)    │
├──────────────────────────────────────────────┤
│  适配器层 (Adapters)                         │
│  Cloudflare ｜ Vercel ｜ Supabase             │
├──────────────────────────────────────────────┤
│  各平台原生 Runtime                           │
│  Workers ｜ Edge Functions ｜ ...             │
└──────────────────────────────────────────────┘
```

### 1. 协议层 — 别人适配我们

OasisWaker 定义了**通用协议标准**，任何想加入网络的平台都需要实现：

- **存储块格式**：块大小、分片策略、元数据规范
- **加密方案**：AES-256-GCM 客户端加密，密钥派生标准
- **节点协议**：注册、心跳、健康检查的接口契约
- **传输接口**：CLI 命令、REST API 端点、WebSocket 事件

**OasisWaker 的协议是标准，平台来适配它。**

### 2. 适配器层 — 我们适配别人

每个平台的 Runtime 不同，我们为每个平台写特定适配器：

- **Cloudflare Workers 适配器** → 用 Workers API 读写 R2
- **Vercel Edge 适配器** → 用 Edge Functions API 读写 Blob
- **Supabase 适配器** → 用 Edge Functions API 读写 Storage

**适配器说平台的语言，但遵守 OasisWaker 的协议。**

### 3. 用户侧 — 用户零适配成本

用户只需要：
1. 安装 CLI
2. OAuth 登录已有平台账号
3. `put` 和 `get`

**用户不需要知道协议、适配器、平台差异——它们只是"存储"。**

---

## 协议优先于实现

| 概念 | 协议定义 | 适配器实现 |
|------|----------|------------|
| 存储 | `put(blockId, data)` / `get(blockId)` | R2 API / Blob API / Storage API |
| 加密 | AES-256-GCM + 密钥派生 | SubtleCrypto / Node crypto |
| 节点 | `register()` / `heartbeat()` / `leave()` | 各平台部署脚本 |
| 发现 | 节点列表 API / WebSocket 事件 | 平台无关（后端实现） |

---

## 层级关系

| 角色 | 行为 |
|------|------|
| **新平台加入网络** | 需要实现 OasisWaker 的适配器接口规范 |
| **OasisWaker 前端/后端** | 对外暴露稳定的 REST API 和 WebSocket 协议 |
| **现有平台 (Cloudflare 等)** | 不变，不知道 OasisWaker 存在 |
| **用户** | 通过 CLI / Dashboard 使用，感知不到平台差异 |

## 架构决策记录

### ADR-001: 协议优先，适配器次之

- **状态**: 已采纳
- **背景**: 初期容易陷入"先写 Cloudflare 适配器"的误区
- **决策**: 先定义协议接口（类型定义、API 契约），再为每个平台实现适配器
- **后果**: 正——新平台加入成本低；负——协议设计需要更多前期思考

### ADR-002: 用户侧零耦合

- **状态**: 已采纳
- **背景**: 用户不应感知到"这个块存在 Cloudflare 上"
- **决策**: 所有平台差异在适配器层封装，CLI 和 Dashboard 只与协议交互
- **后果**: 正——用户体验一致；负——调试时需要适配器层日志

### ADR-003: 适配器可插拔

- **状态**: 已采纳
- **背景**: 用户可能只用一个平台，也可能用三个
- **决策**: 适配器是独立模块，通过统一接口注册到协议层
- **后果**: 正——按需加载，减少 bundle；负——增加模块化复杂度

---

## 对"适配"的澄清

**不是"别人适配我们"**——因为现有平台（Cloudflare、Vercel、Supabase）不知道 OasisWaker 的存在，也不需要知道。

**不是"我们适配别人"**——因为我们不是为每个平台重写一套产品，而是写适配器把它们的能力纳入同一网络。

**正确理解**：OasisWaker 定义协议，适配器翻译协议到平台 API。适配器是翻译层，不是适配层。
