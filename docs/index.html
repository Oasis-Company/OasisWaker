OasisWaker is a decentralized edge storage network. Two subsystems coexist: the v1.0 CLI (published, Node.js) handles OAuth login and platform adapter deployment; the v2.0 backend (FastAPI, in development) provides a central coordination server with REST API, WebSocket, multi-tenant database, and a Next.js dashboard.

System Architecture

Two subsystems. One protocol.

OasisWaker defines a block storage protocol. Adapters translate it to each platform's native API. The v1.0 CLI handles authentication and deployment; the v2.0 server provides coordination, metadata, and real-time events.

v1.0 — Published

CLI Tool

Node.js 18+ CLI with Commander.js. OAuth2.0 PKCE login to Cloudflare, Vercel, and Supabase. Deploys edge workers that implement the block storage protocol. Credentials encrypted with AES-256-GCM.

Tutorial

Local setup guide.

From zero to a running backend with dashboard. Requires Python 3.10+, Node.js 18+, and pnpm.

1

Clone and install backend dependencies

Using Poetry (recommended) or pip. The backend uses FastAPI with SQLAlchemy async and aiosqlite for development.

# Backend setup (Python 3.10+ required) cd backend # Install with poetry poetry install # Or install with pip pip install fastapi uvicorn sqlalchemy aiosqlite \ pydantic-settings python-jose passlib[bcrypt] \ alembic limits httpx python-dotenv # Copy environment config cp .env.example .env # or let defaults apply
2

Start the backend server

Uvicorn with auto-reload. The server initializes the SQLite database and creates all tables on startup. Swagger docs at /docs.

# Start the FastAPI server uvicorn app.main:create_app --factory \ --host 0.0.0.0 --port 8000 --reload # Or with poetry poetry run uvicorn app.main:create_app \ --factory --host 0.0.0.0 --port 8000 --reload # Verify curl http://localhost:8000/health # {"status":"ok","service":"oasiswaker-v2"}
3

Create a user and get an API key

Sign up, then create an API key for programmatic access. The API key format is ow_<5chars>_<40chars>.

# Sign up curl -X POST http://localhost:8000/api/v1/auth/signup \ -H "Content-Type: application/json" \ -d '{"email":"demo@test.com","password":"test123"}' # Create API key (use the access_token from signup) curl -X POST http://localhost:8000/api/v1/auth/api-keys \ -H "Authorization: Bearer <access_token>" \ -H "Content-Type: application/json" \ -d '{"name":"dev-key"}' # {"id":"...","key_prefix":"ow_abcde",\ # "full_key":"ow_abcde_..."}
4

Register a node and report heartbeat

Nodes are the core entity. Each node has a unique node_id, status, platform, and storage tracking.

# Register a node curl -X POST http://localhost:8000/api/v1/nodes \ -H "Authorization: Bearer <token>" \ -H "Content-Type: application/json" \ -d '{"name":"my-first-node",\ "node_id":"node-001",\ "platform":"cloudflare",\ "version":"2.0.0"}' # Send heartbeat curl -X POST \ http://localhost:8000/api/v1/nodes/<id>/heartbeat \ -H "Authorization: Bearer <token>" \ -H "Content-Type: application/json" \ -d '{"status":"online","used_storage":0}' # List nodes curl http://localhost:8000/api/v1/nodes \ -H "Authorization: Bearer <token>"
5

Start the frontend dashboard

Next.js 14 App Router with Swiss Style design. Connects to localhost:8000 by default.

# Frontend setup (Node.js 18+ required) cd frontend pnpm install pnpm dev # Open http://localhost:3000 # Login with the credentials from step 3
6

Run database migrations (Alembic)

Schema changes are managed via Alembic. The current schema has two migrations: initial table creation and foreign key addition.

# Run migrations cd backend alembic upgrade head # Check history alembic history # Create a new migration alembic revision --autogenerate \ -m "description"
7

CLI tool: OAuth login to a platform

The v1.0 CLI handles OAuth2.0 PKCE login to Cloudflare, Vercel, or Supabase, then deploys edge workers.

# Install the CLI npx @oasisbio/oasiswaker init # Login to a platform (opens browser for OAuth) oasiswaker login cloudflare # Deploy the adapter worker oasiswaker deploy cloudflare # Check status oasiswaker status
Design Decisions

Key constraints.

Hard-coded in the project memory and enforced at every layer.

All endpoints authenticated

Every API endpoint (except /health, /signup, /login) requires JWT Bearer or API Key. The get_current_user dependency is injected at the router level.

Row-level security

All queries are scoped to the authenticated user via scoping.scope_query_by_user(). A user can only see their own nodes, blocks, and API keys.

Rate limiting enforced

Sliding window rate limiter: 60 req/min for authenticated users, 20 req/min for unauthenticated. In-memory storage (Redis planned for production).

Architecture is documented. Code is real.
